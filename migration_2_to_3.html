迁移常见问题解答（2.x -> 3.0）#

危险

这个指南仍在制作中。

这个版本引入了许多重大变化和架构改进。它有助于减少代码中全局变量的数量，提供了有用的机制来模块化代码，并通过PyPI上的软件包来创建可共享的模块。它还使中间件和过滤器更易控制，还有其他改进。

在这个页面上，你可以阅读关于与上一个稳定的 2.x 版本相关的更改。

备注

这个页面更像是一个详细的变更日志，而不是一个迁移指南，但它将在将来进行更新。

随时欢迎对此页面进行贡献，如果你发现有遗漏的内容。

调度器#

  • Dispatcher 类现在不再在其初始化程序中接受 Bot 实例。相反，Bot 实例应该只在调度程序中用于启动轮询或处理来自 Webhook 的事件。这种方法还允许同时使用多个机器人实例（“多机器人”）。

  • Dispatcher 现在可以通过另一个类似于Dispatcher的东西进行扩展，名为 Router (阅读更多 »)。

  • 使用路由，您可以轻松模块化您的代码，并且有可能在不同项目之间共享这些模块。

  • 从所有事件处理程序装饰器和注册方法中去除了 _handler 后缀。 (阅读更多»)

  • 执行程序已完全删除；现在可以直接使用调度程序开始轮询API或处理来自API的webhook。

  • 限流方法已完全移除; 现在可以使用中间件来控制执行上下文并实现任何所需的限流机制。

  • 从API类型Bot和Dispatcher对象中移除了全局上下文变量。 从现在开始，如果您想在处理程序或过滤器中访问当前的bot实例，您应该接受参数 bot: Bot 并使用它来替代 Bot.get_current() 。 在中间件中，可以通过 data["bot"] 访问。

  • 为了跳过待定的更新，您现在应该直接调用 aiogram.methods.delete_webhook.DeleteWebhook 方法，而不是将 skip_updates=True 传递给启动轮询方法。

过滤事件#

  • 关键字过滤器不再能够使用；请明确使用过滤器。 (阅读更多 »)

  • 由于关键词过滤器被移除，所有先前默认启用的过滤器（例如状态和内容类型）现在被禁用了。如果您希望使用它们，必须明确指定它们。例如，不要使用 @dp.message_handler(content_types=ContentType.PHOTO) ，而应该使用 @router.message(F.photo)

  • 大多数常见的过滤器都已被“魔法过滤器”取代。 (阅读更多 »)

  • 默认情况下，消息处理程序现在接收任何内容类型。如果你想要特定的，只需添加适当的过滤器（Magic或任何其他）。

  • 状态过滤器不再默认启用。这意味着如果在 v2 中使用了 state="*" ，则在 v3 中不应该传递任何状态过滤器。相反，如果在 v2 中没有指定状态，则现在需要在 v3 中指定。

  • 为每个路由器添加了注册全局过滤器的可能性，这有助于减少代码重复，提供了更容易控制每个路由器目的的方式。

机器人 API#

  • 所有的API方法现在都是具有验证的类，通过 pydantic 实现。这些API调用也作为Bot类中的方法可用。

  • 已添加更多预定义的枚举并移至 aiogram.enums 子包。例如，现在聊天类型枚举是 aiogram.enums.ChatType 而不是 aiogram.types.chat.ChatType 。

  • HTTP客户端会话已被分离成一个可以在应用程序中不同Bot实例之间重复使用的容器。

  • API异常不再按特定消息分类，因为Telegram没有记录的错误代码。然而，所有错误都按HTTP状态代码分类，并且对于每种方法，只能将一种错误类型与特定代码关联起来。因此，在大多数情况下，您应该仅检查错误类型（按状态代码），而不检查错误消息。

中间件#

  • 中间件现在可以控制执行环境，例如使用上下文管理器。 (阅读更多»)

  • 所有上下文数据现在在中间件、过滤器和处理程序之间进行端到端共享。例如，现在您可以轻松地将一些数据传递到中间件内的上下文中，并且可以通过关键字参数在过滤器层以与处理程序相同的方式获取它。

  • 添加了名为 flags 的机制，可在中间件配合下帮助定制处理程序行为。 (阅读更多 »)

键盘标记#

  • 现在 aiogram.types.inline_keyboard_markup.InlineKeyboardMarkup 和 aiogram.types.reply_keyboard_markup.ReplyKeyboardMarkup 不再具有用于扩展的方法，而是必须分别使用标记生成器 aiogram.utils.keyboard.ReplyKeyboardBuilder 和 aiogram.utils.keyboard.KeyboardBuilder (阅读更多 »)

回调数据#

  • 回调数据工厂现在使用 pydantic 模型严格类型化。 (阅读更多»)

有限状态机#

  • 状态过滤器将不再自动添加到所有处理程序中；如果您想使用它，您将需要指定状态。

  • 增加了改变FSM策略的可能性。 例如，如果你想基于聊天话题而不是聊天中的用户来控制每个用户的状态，你可以在调度程序中指定这一点。

  • 现在 aiogram.fsm.state.State 和 aiogram.fsm.state.StateGroup 没有像 .set() ， .next() 等帮助方法。

  • 相反，您应该直接将它们传递给 aiogram.fsm.context.FSMContext 来设置状态(点击了解更多»)

  • 状态代理已弃用；您应该分别通过调用 state.set_data(...) 和 state.get_data() 来更新状态数据。

发送文件#

  • 从现在开始，发送文件时应该先将文件包装在一个InputFile对象中， 而不是直接将IO对象传递给API方法。(Read more »)

网络钩子#

  • aiohttp 网络应用程序配置已经简化。

  • 默认情况下，当您 响应更新 请求时，已添加上传文件的功能（仅适用于 webhook）。

Telegram API 服务器#

  • server 参数已从 Bot 实例移动到 api 在 BaseSession 中。

  • 常量 aiogram.bot.api.TELEGRAM_PRODUCTION 已移动至 aiogram.client.telegram.PRODUCTION 。
